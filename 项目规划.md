# AI日报项目规划

> 创建时间：2026-05-15
> 更新时间：2026-05-15 v2.0
> 状态：架构设计中

---

## 一、项目定位

**全自动AI日报网站**——对标橘鸦AI早报的信息质量，实现从采集到发布的全链路自动化。

- **核心**：信息源自动采集 + AI智能整理 + 网站自动发布
- **原则**：全自动，零人工干预
- **暂不含视频**（后期扩展）
- **独立项目**，与新叶AI无关

## 二、参考项目：橘鸦AI早报

> 本项目以橘鸦AI早报为主要参考，以下信息来自对其GitHub仓库和微信文章的调研，供开发团队直接使用，无需再次爬取。

### 2.1 项目简介

橘鸦AI早报（https://github.com/imjuya/juya-ai-daily）是一个**AI辅助生成的每日AI资讯摘要项目**，每日自动采集AI领域资讯，生成Markdown日报并通过GitHub Pages发布，同时制作视频发布到B站和YouTube。

- **主站**：https://imjuya.github.io/juya-ai-daily/
- **RSS**：https://imjuya.github.io/juya-ai-daily/rss.xml
- **视频工具**：https://github.com/imjuya/md2video（Markdown转视频）

### 2.3 橘鸦核心流程

1. **采集**：`collect.py` 从RSS/Twitter/GitHub/博客等源采集AI资讯
2. **AI处理**：调用LLM API（硅基流动）生成结构化摘要、分类
3. **内容管理**：以GitHub Issues为CMS，每期日报一个Issue，用标签（Top/Friends/TODO等）分类
4. **发布**：`main.py` 读取Issues生成README.md和RSS，Zola构建静态站部署到GitHub Pages
5. **视频**：`md2video`工具链：Markdown → TTS音频 → HTML截图 → MoviePy合成视频

### 2.4 橘鸦日报内容格式

```
标题: AI 早报 YYYY-MM-DD

## 概览
### 要闻
- 新闻1 ↗ #1
- 新闻2 ↗ #2

### 模型发布
- 新闻3 ↗ #3

## [新闻标题] #编号
[摘要/blockquote]
[详细内容]
[图片]
相关链接：[原文链接]
```

| 要闻 | 重大突破、重磅发布 |
| 模型发布 | 新模型、模型更新 |
| 开发生态 | Agent、SDK、工具、框架 |
| 产品应用 | 产品功能、用户端更新 |
| 技术与洞察 | 技术解析、研究报告 |
| 行业动态 | 融资、上市、政策 |
| 前瞻与传闻 | 传闻、预告、规划 |

### 2.6 橘鸦技术栈

| 层面 | 技术 |
|------|------|
| 采集 | Python + requests + BeautifulSoup + feedparser |
| AI | OpenAI SDK + 硅基流动API |
| CMS | GitHub Issues |
| 站点生成 | Zola (Rust静态站生成器) + even主题 |
| 部署 | GitHub Actions + GitHub Pages |
| 视频制作 | md2video (TTS+Screenshot+MoviePy+FFmpeg) |

### 2.7 橘鸦的优劣势分析

**优势（值得借鉴）**：
- GitHub Issues做CMS，零成本、版本控制、天然CI集成
- 双阶段CI/CD：内容生成→站点部署，关注点分离
- 编号系统：每条新闻唯一编号，便于引用
- 概览+详情结构：快速浏览+深度阅读
- 统一内容源，多渠道输出

**劣势（我们改进的方向）**：
- **半自动**：采集后需人工审核/编辑Issue，不是全自动
- **数据非结构化**：Markdown文件存储，无法高效搜索/筛选/统计
- **交互有限**：静态展示，无搜索、分类筛选、订阅推送
- **站点简陋**：Zola静态主题，UI体验一般
- **视频流程重**：Selenium截图+MoviePy，依赖Chrome，耗时且不稳定

### 3.1 架构设计原则

1. **模块解耦**：采集、处理、生成、发布四层独立，任一层可单独替换/升级
2. **数据驱动**：层与层之间通过结构化数据流转，不依赖文件系统
3. **容错优先**：单个信息源失败不影响整体，处理链路单步失败可跳过
4. **可观测性**：每层有日志和状态输出，便于排查问题

## 四、模块详细设计

### 4.1 采集层（Collector）

**职责**：从多种信息源自动获取最新AI资讯，统一格式输出。

#### 信息源分级

**一级源（RSS自动采集，最稳定）**

| 来源 | 类型 | 优先级 | 说明 |
|------|------|--------|------|
| OpenAI Blog | RSS | SSS | 官方动态 |
| Google AI Blog | RSS | SSS | Gemini等 |
| Anthropic Blog | RSS | SSS | Claude相关 |
| Meta AI Blog | RSS | SS | Llama等 |
| DeepSeek | RSS/API | SS | 国产头部 |
| 智谱AI | RSS/Web | SS | GLM系列 |
| Hugging Face Blog | RSS | SS | 开源生态 |
| xAI Blog | RSS | SS | Grok |
| 机器之心 | RSS | S | 中文AI媒体 |
| 量子位 | RSS | S | 中文AI媒体 |
| 36kr AI频道 | RSS | S | 产业动态 |
| TechCrunch AI | RSS | S | 英文科技 |
| The Verge AI | RSS | S | 英文科技 |

**二级源（需要爬取或API，稳定性一般）**

| 来源 | 方式 | 优先级 | 说明 |
|------|------|--------|------|
| Twitter/X AI KOL | API | SS | 核心人物动态 |
| 微信公众号 | 爬取/RSSHub | S | 国内深度文章 |
| Reddit r/MachineLearning | RSS | A | 社区讨论 |
| Product Hunt AI | API | A | 新产品 |
| ArXiv cs.AI | RSS | A | 学术论文 |
| GitHub Trending | API | A | 开源项目 |

#### 采集策略

- **频率**：每30分钟采集一次一级源；二级源每小时一次
- **增量采集**：记录上次采集时间，只拉取新内容
- **格式统一**：所有源输出统一结构（标题、正文/摘要、URL、来源、时间）
- **容错**：单源超时3秒跳过，连续3次失败标记为不可用

### 4.2 处理层（Processor）

**职责**：对原始资讯进行去重、摘要、分类、打分，输出结构化新闻条目。

#### 处理Pipeline

```
原始资讯 → 去重 → AI摘要 → 关键词提取 → 分类 → 打分 → 筛选
```

**去重**（三层机制）：
1. URL去重：相同链接直接跳过
2. 标题去重：相似度 > 0.85 视为重复
3. 语义去重：AI判断是否与72小时内已有新闻描述同一事件，保留信息最完整的

**AI摘要**：
- 输入：原始文章文本
- 输出：50-100字结构化摘要（核心事实+关键数据+影响范围）
- LLM选型：DeepSeek V4 Flash（性价比）或 MiMo API（备选）

**自动分类**（参考橘鸦体系）：

| 分类 | 编号 | 说明 |
|------|------|------|
| 要闻 | 01 | 重大突破、重磅发布 |
| 模型发布 | 02 | 新模型、模型更新 |
| 开发生态 | 03 | Agent、SDK、工具、框架 |
| 产品应用 | 04 | 产品功能、用户端更新 |
| 技术与洞察 | 05 | 技术解析、研究报告 |
| 行业动态 | 06 | 融资、上市、政策 |
| 开源项目 | 07 | 开源发布、重大更新 |
| 前瞻与传闻 | 08 | 传闻、预告、规划 |

**重要度打分**：

| 维度 | 说明 |
|------|------|
| 影响力 | 影响多少人/行业（1-10） |
| 时效性 | 是否刚发生（1-10） |
| 独特性 | 是否独家/首次（1-10） |
| 可靠性 | 信源可信度（1-10） |

总分 → 映射等级：

| 总分 | 等级 | 处理 |
|------|------|------|
| ≥35 | SSS | 置顶要闻 |
| 25-34 | SS | 重要进展 |
| 15-24 | S | 值得关注 |
| 8-14 | A | 一般资讯 |
| <8 | B | 过滤掉 |

### 4.3 生成层（Generator）

**职责**：将处理后的结构化新闻条目组装为完整的日报。

- 按分类排序，同分类内按等级降序
- AI生成每个分类的导语（1-2句概述）
- 每条新闻2-3句精炼摘要
- 输出：Markdown（人读）+ JSON（机读，供前端消费）

### 4.4 发布层（Publisher）

**职责**：将生成的日报持久化并对外发布。

- **数据库写入**：日报元数据 + 新闻条目入库
- **网站更新**：触发Vercel重新构建（或ISR增量更新）
- **RSS输出**：生成标准RSS 2.0 Feed
- **推送通知**（后期）：邮件/Webhook通知订阅者

### 4.5 调度层（Scheduler）

**职责**：控制各模块的执行时机和频率。

| 任务 | 频率 | 说明 |
|------|------|------|
| 信息采集 | 每30分钟 | 采集所有活跃源 |
| 去重检查 | 每日06:30 | 跨天去重 |
| 日报生成 | 每日07:00 | 汇总生成当日日报 |
| 日报发布 | 每日08:00 | 发布到网站+RSS |
| 源健康检查 | 每周日 | 检查源可用性 |

调度方式：GitHub Actions Cron（采集/处理）+ Vercel Cron（发布）

## 五、技术选型

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| 采集 | Python + feedparser + requests | 成熟方案，橘鸦也用Python，生态丰富 |
| 处理 | Python + LLM API | AI处理自然语言，Python调用LLM API最方便 |
| 存储 | Supabase (PostgreSQL) | 免费额度够用，自带REST API，与新叶AI技术栈一致 |
| 前端 | Next.js 16 + Tailwind CSS | SSG性能优先，团队熟悉 |
| 部署 | Vercel + GitHub Actions | 自动构建+定时任务，零运维 |
| RSS输出 | Python feedgen | 标准RSS 2.0生成 |

**LLM选型**：
- 主力：DeepSeek V4 Flash（性价比高，中文能力强）
- 备选：MiMo API（免费额度，冰糖音色同源）

## 七、与橘鸦的关键差异

| 维度 | 橘鸦 | 本项目 |
|------|------|--------|
| 采集 | 半自动（部分手动筛选） | **全自动** |
| 发布 | GitHub Issues + Pages | Supabase + Vercel |
| 交互 | 静态展示 | 搜索+分类筛选+订阅 |
| 数据 | Markdown文件存储 | 结构化数据库 |
| 视觉 | Zola静态主题 | Next.js现代UI |
| 去重 | 人工判断 | AI语义去重 |
| 视频 | 有（md2video） | 暂无（后期扩展） |
